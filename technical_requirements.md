# Технические требования: инвестиционный AI-помощник

Связанный бизнес-документ: [business_requirements.md](business_requirements.md).

Связанный MCP-контракт: [mcp_server_api_design.md](mcp_server_api_design.md).

## 1. Назначение документа

Документ описывает технические требования к первому релизу инвестиционного AI-помощника: интеграцию с Tinkoff Invest API, получение и кэширование данных, модель данных, расчеты, безопасность и технические ограничения.

Бизнес-цели, пользовательские сценарии, MVP и критерии успеха описаны отдельно в [business_requirements.md](business_requirements.md).

## 2. Основной технический контур

Первый релиз реализуется на Python. Это упрощает интеграцию с Tinkoff Invest API, OpenAI API и официальным MCP Python SDK.

Основной технический контур строится вокруг Tinkoff Invest API и Python-клиента `invest-python`.

Официальная документация:

- Tinkoff Invest API: https://tinkoff.github.io/investAPI/
- Python-клиент `invest-python`: https://tinkoff.github.io/invest-python/
- Примеры `invest-python`: https://tinkoff.github.io/invest-python/examples/

Технические принципы:

- API является основным источником данных о портфеле;
- все данные, полученные из API, кэшируются локально;
- сырые ответы API желательно хранить отдельно от нормализованной модели;
- расчеты должны быть воспроизводимыми;
- торговые операции через API не входят в MVP;
- приложение должно работать в режиме чтения данных.

## 3. Интеграция с Tinkoff Invest API

### 3.1. Что нужно получать через API

Система должна получать:

- список счетов пользователя;
- текущие позиции по счетам;
- текущее портфолио со статистикой доходности и стоимости активов;
- историю операций с постраничной загрузкой;
- брокерские отчеты для сверки истории и спорных случаев;
- доступные денежные и валютные позиции;
- справочник инструментов;
- данные по акциям, облигациям, фондам, валютам и кэшу;
- текущие и исторические рыночные цены;
- торговые статусы инструментов;
- стримы портфеля и позиций, если они понадобятся для более свежего обновления.

### 3.2. Счета

Система должна:

- получать список брокерских счетов;
- позволять выбрать счета, которые участвуют в анализе;
- хранить `account_id`, тип счета, название счета и статус счета;
- уметь показывать данные по каждому счету отдельно;
- уметь агрегировать выбранные счета в единый портфель.

### 3.3. Портфель и позиции

Система должна получать и хранить:

- текущую стоимость портфеля;
- позиции по инструментам;
- валютные и денежные позиции;
- количество инструмента;
- текущую цену;
- текущую стоимость позиции;
- доходность, если она возвращается API;
- долю позиции в портфеле;
- признак блокировки инструмента или позиции, если он возвращается API.

### 3.4. История операций

Система должна получать историю операций из API и учитывать, что история может быть технически сложной.

Требования:

- поддерживать постраничную загрузку операций;
- хранить идентификаторы операций из API;
- хранить дату, тип операции, инструмент, количество, цену, валюту, комиссию и счет;
- учитывать, что идентификаторы операций могут меняться;
- учитывать, что после корпоративных действий история по инструменту может быть неполной или неоднозначной;
- предусмотреть сверку с брокерским отчетом для спорных случаев.

### 3.5. Брокерские отчеты

Брокерский отчет нужен как источник сверки, если данные операций из API недостаточны или вызывают расхождения.

Система должна предусмотреть:

- загрузку или получение брокерского отчета через API;
- хранение отчета как отдельного источника данных;
- сверку операций, комиссий, выплат и остатков;
- отображение расхождений между API-операциями и отчетом.

## 4. Поддерживаемые инструменты

Первый релиз поддерживает только российский рынок.

Классы активов:

- акции;
- облигации;
- фонды;
- валюта;
- кэш.

Для каждого инструмента нужно хранить:

- тикер;
- FIGI;
- UID или другой внутренний идентификатор API;
- ISIN, если доступен;
- название;
- класс актива;
- валюта;
- торговая площадка;
- сектор или отрасль, если доступно;
- лотность;
- торговый статус;
- признак доступности для торгов;
- дополнительные параметры для облигаций.

Для облигаций дополнительно нужны:

- дата погашения;
- купонная ставка;
- график купонов;
- номинал;
- валюта номинала;
- амортизация, если применимо;
- оферта, если применимо;
- кредитный рейтинг эмитента или выпуска, если доступен из внешних источников.

## 5. Кэширование данных

Все данные, полученные из API, должны кэшироваться локально.

Для каждой записи в кэше нужно хранить:

- источник данных;
- метод API или тип загрузки;
- дату и время получения;
- исходные параметры запроса;
- сырые данные ответа, если это не нарушает ограничения безопасности;
- нормализованную запись;
- признак актуальности;
- ошибку загрузки, если запрос завершился неуспешно.

Кэш нужен для:

- ускорения повторных запросов;
- воспроизводимости расчетов;
- возможности отвечать на часть вопросов при временной недоступности API;
- сравнения текущего состояния с предыдущими снимками;
- аудита рекомендаций и отчетов.

Система должна явно различать:

- свежий ответ API;
- данные из кэша;
- устаревшие данные;
- неполные данные.

## 6. Обновление данных

Система должна поддерживать несколько типов обновления:

- первичная синхронизация;
- ручное обновление по запросу пользователя;
- фоновое плановое обновление;
- частичное обновление справочников;
- частичное обновление цен;
- обновление истории операций за период;
- обновление через стримы портфеля и позиций в будущих версиях.

При обновлении система должна:

- не терять уже загруженные данные;
- не дублировать операции;
- фиксировать время последней успешной синхронизации;
- фиксировать время последней неуспешной синхронизации;
- показывать пользователю, если данные устарели.

## 7. Нормализованная модель данных

Минимальные сущности:

- пользователь;
- брокерский счет;
- инструмент;
- позиция;
- снимок портфеля;
- операция;
- денежная позиция;
- рыночная цена;
- свеча;
- корпоративное событие;
- новость;
- риск-сигнал;
- рекомендация;
- отчет;
- чат-сообщение.

### 7.1. Позиция

Позиция должна содержать:

- счет;
- инструмент;
- количество;
- среднюю цену, если ее можно корректно рассчитать;
- текущую цену;
- текущую стоимость;
- валюту;
- долю в портфеле;
- доходность;
- дату и источник последнего обновления.

### 7.2. Снимок портфеля

Снимок портфеля должен содержать:

- дату и время;
- список счетов;
- общую стоимость;
- стоимость по классам активов;
- стоимость по валютам;
- стоимость по секторам;
- список позиций;
- рассчитанные риск-метрики;
- источник данных.

### 7.3. Рекомендация

Рекомендация должна содержать:

- дату;
- контекст портфеля;
- действие;
- инструмент или класс активов;
- сумму или долю, если применимо;
- обоснование;
- связь с целью пользователя;
- риски;
- альтернативы;
- уровень уверенности;
- использованные данные;
- статус выполнения;
- результат спустя время, если его можно оценить.

## 8. Расчеты и аналитика

Система должна рассчитывать:

- структуру портфеля по классам активов;
- структуру по инструментам;
- структуру по эмитентам;
- структуру по секторам;
- структуру по валютам;
- вклад активов в изменение портфеля;
- отклонение от целевой аллокации;
- концентрацию в одном активе;
- концентрацию в одном эмитенте;
- концентрацию в одном секторе;
- концентрацию в одной валюте;
- базовые риск-сигналы.

Расчеты должны быть воспроизводимы:

- использовать сохраненный снимок портфеля;
- ссылаться на дату и источник цен;
- хранить версию алгоритма или расчетного правила, если оно меняется;
- не смешивать свежие и устаревшие данные без явной маркировки.

## 9. Новости и внешние источники

Tinkoff Invest API покрывает брокерские и рыночные данные, но не закрывает все потребности по новостям, рейтингам и фундаментальной аналитике.

Нужно отдельно выбрать источники для:

- новостей по российским эмитентам;
- финансовой отчетности;
- кредитных рейтингов;
- дивидендных объявлений;
- купонных событий;
- оферт и погашений;
- санкционных и регуляторных событий;
- макроэкономических данных;
- ключевой ставки;
- отраслевых новостей.

Для каждого внешнего источника нужно хранить:

- название источника;
- ссылку или идентификатор материала;
- дату публикации;
- дату события;
- связанные инструменты и эмитенты;
- краткую выжимку;
- оценку важности для портфеля.

## 10. Выходные артефакты

Система должна формировать следующие типы выходов:

- ответ в чате;
- краткий ежедневный обзор;
- недельный отчет;
- месячный отчет;
- срочное уведомление;
- исследование по активу;
- риск-сигнал;
- рекомендация по действию.

Каждый аналитический ответ должен содержать:

- краткий вывод;
- данные, на которых основан вывод;
- связь с портфелем;
- связь с целью пользователя;
- риски;
- уровень уверенности;
- дату актуальности данных.

## 11. Безопасность

Требования:

- токен Tinkoff Invest API должен храниться безопасно;
- токен не должен попадать в логи, отчеты, сообщения чата или репозиторий;
- доступ к данным портфеля должен быть ограничен авторизованным пользователем;
- данные портфеля должны храниться безопасно;
- секреты и брокерские ключи не должны храниться в открытом виде;
- система должна работать в режиме чтения данных;
- торговые методы API не должны использоваться в MVP;
- если торговые функции появятся в будущем, каждое действие должно требовать явного подтверждения пользователя.

Желательно предусмотреть отдельный режим "read-only", в котором приложение технически не вызывает методы выставления заявок.

## 12. Надежность и ошибки

Система должна:

- показывать дату и время последнего обновления данных;
- сообщать, если часть данных недоступна;
- сообщать, если данные устарели;
- корректно обрабатывать временную недоступность API;
- корректно обрабатывать пустые ответы;
- корректно обрабатывать частично загруженную историю;
- продолжать отвечать на часть вопросов из кэша;
- явно маркировать ответы, построенные на кэшированных данных.

Ошибки API должны логироваться без токенов и чувствительных данных.

## 13. Технические ограничения MVP

В MVP не требуется:

- выставление торговых заявок;
- автоматическое совершение сделок;
- поддержка иностранных брокеров;
- поддержка международных рынков;
- налоговая аналитика;
- импорт CSV/XLSX как основной способ загрузки;
- полноценный модуль исполнения рекомендаций;
- сложное стресс-тестирование;
- высокочастотные обновления.

## 14. Открытые технические вопросы

- Где хранить локальный кэш: SQLite, Postgres, файловое хранилище или другой вариант?
- Какой интерфейс будет первым: Telegram, веб, оба варианта или другой формат?
- Какой механизм фоновых задач использовать для обновления данных?
- Какие внешние источники выбрать для новостей, рейтингов и отчетности?
- Нужно ли хранить сырые API-ответы целиком или только нормализованную модель?
- Какие правила устаревания кэша нужны для разных типов данных?
- Нужен ли отдельный технический запрет на вызов торговых методов API?
- Как обрабатывать расхождения между API-операциями и брокерским отчетом?

### 14.1. Принятые технические решения (по факту реализации)

- Хранилище — **SQLite** (`storage.py`, путь в `INVESTOR_MCP_STORAGE_PATH`): профиль,
  выбранные счета, статус синхронизации, снимки портфеля, рекомендации, отчёты и кэш
  состава.
- Интерфейс — **remote MCP-коннектор** к ChatGPT/Claude (без собственного фронта),
  публикуется через Cloudflare Tunnel. Деплой и эксплуатация — [DEPLOYMENT.md](DEPLOYMENT.md).
- Кэш состава портфеля (позиции, счета) — **TTL 1 сутки** (`INVESTOR_MCP_CACHE_TTL_SECONDS`,
  по умолчанию 86400), in-memory + персист в SQLite. Тулы состава читают из кэша; брокер
  опрашивается раз в сутки или по `investor_sync_data` / `get_portfolio(refresh=true)`.
  Статус данных маркируется `data_status`: `fresh` / `cached` / `stale`.
- Read-only гарантируется на уровне адаптера: торговые методы Tinkoff не вызываются
  вовсе; дополнительно доступна bearer-авторизация и секретный путь.
- Оценка стоимости приведена к базовой валюте (FX-конвертация) с учётом НКД по облигациям;
  совпадает с брокерским итогом.
- Сырые ответы API целиком пока **не** сохраняются — кэшируется нормализованная модель
  (снимки портфеля + кэш состава). Полное хранение сырых ответов и кэш операций/новостей —
  следующий шаг.
- Внешние источники новостей/рейтингов (OpenAI/web search) пока не подключены;
  `investor_get_news_digest` отдаёт mock-ленту.
